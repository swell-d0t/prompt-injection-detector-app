01 Overview:

Prompt injection attacks are malicious inputs designed to hijack or manipulate large language models. This project fine-tunes a three classical ML models-- Linear Regression, Random Forest, and SVM-- and a DistilBERT transformer to detect these attacks, then stress-tests it. The models are tested against both synthetic and real adversarial evasion techniques.

Research question: How robust is a transformer-based prompt injection detector against attacks specifically designed to evade it, and what does that reveal about the role training data plays in that robustness?

02 How the Project Evolved:

01

A Supposedly Near-Perfect Model (DistilBERT)

DistilBERT fine-tuned on a synthetic, LLM-generated prompt injection dataset scored close to 100% detection on clean test data, which matched a simple logistic regression baseline almost exactly.

02

Robustness Testing Revealed A Blind Spot

Against real adversarial evasion techniques, character-level Unicode manipulation (invisible characters, homoglyphs, unusual spacing) consistently evaded detection almost entirely.

0% detection on Unicode Tags Smuggling and Spaces attacks
03

Diagnosed as a Data Problem, Not Just a Model Problem

Matching accuracy between a transformer and a linear model on clean data was itself a clue: the synthetic training data was likely too narrow, encouraging shortcut learning rather than genuine robustness.

04

Retrained on Real, Human-Generated Attacks

Fine-tuning on the Tensor Trust dataset: 280,000+ distinct adversarial prompts submitted by real players of an online security game. This dataset effectively closed nearly all of the Unicode blind spot, with no preprocessing fix required.

Detection on Spaces, Unicode Tags Smuggling, and Homoglyphs rose from 0–9% to 100%

Inputs & outputs: every model takes a single raw text prompt as input. Classical models (Logistic Regression, SVM, Random Forest) convert it into numeric features via TF-IDF or sentence embeddings before classifying. DistilBERT tokenizes it directly. All five output a binary label — safe or injection detected — alongside a confidence score.

Datasets used: xTRam1/safe-guard-prompt-injection (primary training data), Tensor Trust (human-generated attacks used to retrain DistilBERT), Mindgard (adversarial evasion robustness testing), and HackAPrompt (independent, held-out generalization testing).

06 Essential Question:

How can the process of creating AI/ML solutions amplify or mitigate bias in the case our group is using?

Bias in a prompt injection detector doesn't show up as a protected-class disparity in the usual sense. It shows up in who gets misclassified, and why. Our first model's training data skewed toward well-documented, synthetically generated attack patterns, which meant it could amplify bias by underperforming on unfamiliar phrasing, informal language, or novel encoding. This caused the flagging of legitimate users unfairly while letting genuinely novel attacks slip through.

We mitigated this by testing against an independently-sourced, human-generated adversarial dataset rather than assuming our synthetic training data was representative. We did this by tracking false positives and false negatives separately, rather than hiding both behind a single aggregate accuracy score. The broader lesson is that a security classifier's fairness problem is concrete, not abstract. An over-flagged legitimate user and an under-caught attacker are both real costs, and responsible development means measuring both rather than optimizing for one.

07 Societal Impact:

Positive

A working, open-source detector gives smaller teams and independent developers who can't afford commercial guardrail products a tested option for protecting LLM-integrated applications from manipulation.

Negative

Any detector this specific is dual-use: the same evaluation that reveals a blind spot is also a roadmap for exploiting it. Over-aggressive detectors also risk silencing legitimate security research and red-teaming discussion that resembles the attacks they're built to catch.

09 Next Steps:

Technical Extensions:

  • Run Random Forest and both Logistic Regression variants through the same Mindgard/HackAPrompt robustness pipeline DistilBERT went through, for a full head-to-head on adversarial robustness.
  • Test whether data augmentation with Unicode-obfuscated training examples closes the remaining word-substitution attack gap.
  • Replace the hand-built homoglyph map with a complete confusables mapping table.

Deployment:

  • Package the DistilBERT + preprocessing pipeline as a lightweight API endpoint other applications could call before passing user input to an LLM.
  • Add usage monitoring to the deployed app to track false positive/negative patterns on real traffic over time.

Research:

  • Control for the non-deterministic training variance observed (from Python's set() hash randomization) via a fixed PYTHONHASHSEED, for reproducible benchmarking.
  • Extend the training-data-diversity finding to additional held-out datasets beyond HackAPrompt.
